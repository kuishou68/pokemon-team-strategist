# 第一轮评审

## 🔴 Critical

### C1. 队伍注入与历史回灌的双重状态，一致性风险
§3 说"每轮注入 ephemeral 队伍消息" + §5.3 说"回灌紧凑摘要含 id"。两套状态并存：
- 注入的队伍上下文（ephemeral，每轮重建）vs 历史里的 tool 消息（持久）
- 若 `replace_team_member` 改了 server team，但历史里旧的 tool 结果还说着旧队伍 → 模型同时看到新注入队伍和旧历史，可能混淆
- **建议**：明确"队伍真相只看注入的当前队伍块"，历史 tool 消息只保留动作摘要不重述完整队伍；注入块用醒目分隔（如 `=== 当前队伍 ===`）并在 system prompt 里声明"以当前队伍块为准"。

### C2. 队伍变更工具的 emit 时序与 team 事件
`add_to_team`/`replace_team_member` 改 server state 后，team 事件何时 emit？若在工具 execute 内 emit，但 agent 循环还可能继续调别的工具 → team 事件可能多次/乱序。
- **建议**：team 事件在每轮工具执行**全部结束后**统一 emit 一次最终快照，而非每个工具各发；或明确"每次 mutation 后立即 emit，前端以最后一次为准"。

## 🟠 Major

### M1. 6 个工具 + 队伍工具，selection 复杂度上升
工具从 A 的 4 个增至 6+（search/type/move/evolution/analyze/recommend + add/remove/replace = 最多 9 个）。小模型在 9 个工具里选对的难度显著高于 4 个。
- **建议**：① 合并——`add/remove/replace` 可合为一个 `update_team(action, ...)`；② 或分阶段暴露（对话意图不涉及队伍时不暴露队伍工具）。需在"工具数量"与"选择准确率"间权衡。

### M2. analyze_team / recommend_switch 的输入来源
这两个工具签名带 `team[]`/`pokemon_names[]` 参数。但队伍已是 server state——让模型再传一遍队伍参数，既冗余又可能传错（模型可能传幻觉队伍）。
- **建议**：这两个工具**不接受队伍参数，直接读 server team state**。模型只需调用，不传队伍。彻底消除模型传错队伍的可能。

### M3. 图片代理的缓存与体积
`/api/sprite/:id` 转发 raw.githubusercontent。若不缓存，等于把"国内访问 GitHub 慢"的问题搬到服务端，每次请求仍慢。
- **建议**：代理必须**磁盘/内存缓存图片**（首次慢，之后快）；或启动时预取常见宝可梦。明确缓存策略，否则代理没解决根本问题。

### M4. 进化链分支的前端布局复杂度
§5 标了分支进化坑，但前端 `EvolutionChain.tsx` 如何画树（伊布 8 分支）没说。横向链好画，树状分支布局复杂。
- **建议**：明确布局方案——分支用纵向堆叠的横向子链，或简化为"列出所有进化去向"的列表式（牺牲视觉但稳）。先定方案避免实施卡壳。

# 第二轮评审（针对规划者裁决后的 v2）

规划者已处理 C1-C2/M1-M4/m1-m2/m4-m5。复审这些修复：

## 🟠 Major

### R1. `update_team(action)` 合并后，schema 表达力下降
合并成单工具用 `action` enum 区分，但不同 action 的必填参数不同（add 要 pokemon，replace 要 target+pokemon，remove 要 target）。单一 schema 无法表达"条件必填" → 校验逻辑要在 validate 里按 action 分支，模型也可能漏参。
- **建议**：`update_team` 的 description 里明确每个 action 需要哪些参数 + 给例子；validate 按 action 分支校验，缺参回结构化 error 让模型补（A 的 F6 经验）。这是合并工具的必要代价，可接受但要写清。

### R2. 每轮注入队伍块会随轮次累积吗？
"ephemeral 每轮重建"——要确保旧的注入块**不残留在历史里**，否则多轮后会有多个互相矛盾的队伍块。
- **建议**：注入块作为**临时 system 消息只在当轮请求拼接，不写入 session.messages**；session 只存真实 user/assistant/tool 消息 + 结构化 team state。明确"注入块永不持久化"。

## 🟡 Minor

| # | 问题 |
|---|---|
| s1 | team 事件"每轮结束统一 emit"——但若一轮内没有队伍变更，应跳过 emit（避免无意义刷新）|
| s2 | `GET /api/team` 与注入块、team 事件三处读同一 server state，确保单一数据源（session.team），三者都从它派生 |
| s3 | typechart 启动时从 PokeAPI 生成 → 启动变慢且依赖网络；建议核对过的静态种子为主，PokeAPI 仅作可选刷新 |
| s4 | update_team 的 server 校验：宝可梦是否存在（防加入幻觉宝可梦）需先 search 确认 |

## ✅ v2 改进确认
- 队伍"以注入块为准 + 历史只存动作摘要"正确解决了 C1 双状态冲突。
- analyze/recommend 不传队伍、直读 state，彻底消除模型传错队伍。
- 工具合并 7 个，比 9 个对小模型友好。

## 第二轮结论
**无 Critical**。R1（合并工具的条件必填）、R2（注入块不持久化）是合并/注入方案的收尾细节，明确即可。s1-s4 实施时注意。方案已接近实施就绪，建议再一轮规划者收尾后即可进入 /plan + 编码。

# 第三轮评审（性能 / 并发 / 数据规模）

前两轮聚焦状态一致性，本轮换"性能与规模"视角。

## 🟠 Major

### T1. recommend_switch 需要对手数据，但对手要现查 PokeAPI → 阻塞评分
对手宝可梦的属性/种族值/速度要先 `GET /pokemon/{opponent}`，再跟队伍 6 只逐一评分。若对手没缓存，整个推荐阻塞在这次网络调用上。
- **建议**：对手查询走 cache；评分纯本地计算（队伍 6 只已在 session.team，含 stats）。明确"只差对手 1 次上游调用"，其余零网络。

### T2. 每轮注入队伍块的 token 成本
6 只队伍 + 每只属性，每轮都注入，多轮后累积 token 不小（虽不持久化，但每轮都发）。配合 A 的 stopWhen=6，一次对话可能注入 6 次。
- **建议**：注入块**精简**（只名+属性+槽位，不含完整种族值/技能）；完整数据在 server state，工具按需取。控制每轮注入 token。

### T3. 进化链 N+1：链上每个宝可梦要单独取图/属性
进化链有 2-3 个形态，每个要 sprite + 可能的属性。一条链可能 3-5 次额外调用。
- **建议**：进化链卡片**懒加载**——先展示名+箭头+条件（链数据一次性来自 evolution-chain），点击某形态才取详情。避免一次拉全链详情。

## 🟡 Minor

| # | 问题 |
|---|---|
| u1 | PokeAPI species/pokemon 是两个端点，搜一只要 2 次调用（pokemon + species 取中文名）→ 并发 Promise.all，不要串行 |
| u2 | typechart 静态种子文件需校验完整性（18×18=324 格）→ 单测断言矩阵维度 + 对称性抽查 |
| u3 | 队伍快照 stats 缓存在 session，但宝可梦数据不变，可全局缓存（不必每 session 重取）|
| u4 | 多语言 flavor_text 有多条（不同游戏版本）→ 取最新或固定版本，避免随机 |
| u5 | recommend_switch 评分权重应可在代码常量集中配置，便于调优 |

## ✅ 确认
- 静态 typechart 本地评分是性能正确选择。
- 官方中文字段避免了 A 的翻译往返开销。

## 第三轮结论
无 Critical。T1-T3 是性能优化（缓存/精简/懒加载），u1-u5 是实现细节。性能方向清晰，进入规划者裁决。
