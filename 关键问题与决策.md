## Q3. 队伍状态如何管理（题目 B 特有问题）

**问题**："队伍"是多轮持久状态，如何管理？

**结论**：**Agent 维护在对话历史里**（不单独建 team 数据结构）：
- 当前队伍体现在对话上下文中，模型从历史读出当前队伍。
- 多轮"换掉水系换电系"：Agent 自主读队伍 + 替换。
- system prompt 明确告知"记住当前队伍（≤6 只），基于现有队伍调整"。
- 前端队伍视图由后端新增的 `team` SSE 事件推送的快照驱动，不要求模型输出结构化 JSON。

## Q7. 加分项深度设计（结合题目 A 教训）

**问题**：结合初步引导和题目 A 经验，深入分析 6 个加分项如何设计。

**贯穿性设计准则（来自题目 A 教训）**：
| A 的教训 | 在 B 加分项上的应用 |
|---|---|
| 功能不能隐藏 | 加分项做成常驻 Dashboard 面板，不塞进聊天流 |
| 正确性靠代码不靠模型 | 克制/对战计算全用静态 typechart |
| Tool + 面板双通道 | 每个加分项既是 Tool 又是常驻视图 |
| 时间轴推演 | view 由 team SSE 事件驱动，数据到位才渲染 |
| 静态词典优先 | 宝可梦名/属性/进化条件静态词典，零延迟 |
| 浏览器实测 | 每个图表/动画真实渲染验证 |

**核心架构决策：Dashboard + Chat 双栏布局**
- 题目 A 最大教训是"功能塞进聊天流被刷走、不可发现"。B 用结构性解法：右栏常驻 Dashboard（队伍+克制+对战助手），永远可见。
- 移动端收为抽屉/标签页。面板操作转为 chat 消息发给 Agent（Agent 为真相源）。

**逐项设计要点**：
1. **队伍视图**：team SSE 驱动；空态显示 6 个空精灵球槽（可发现性）；点 X = 发"换掉第 N 只"给 Agent；grid items-start + pokemon.id 稳定 key。
2. **克制覆盖分析**：静态 18×18 typechart 代码算（防幻觉）；防御+进攻双维度；短板检测（无人抵抗且无人克制→标红）；简单图标↔详细表格双视图；analyze_team Tool + 常驻面板双通道。
3. **种族值雷达图**：自绘 SVG（不引图表库）；单只雷达 + 队伍柱状对比；游戏风 HP 条上色；aria-label 带数值。
4. **进化链**：⚠️ **分支进化坑**（伊布 8 分支、性别/道具/亲密度条件）——进化链是树结构非线性，规范化与布局都要支持分支（B 版的"filter.php 字段差异"）；进化条件走静态词典翻译。
5. **对战模拟建议（重新纳入）**：之前排除，重新评估后**纳入**——最体现策略师定位，且静态计算性价比高。不做完整伤害公式，做 `recommend_switch(opponent)`：代码对每只队员算防御/进攻/速度三项加权评分（属性 60%/速度 20%/种族值 20%，透明可调）→ 排序推荐 + 理由；代码计算防恐慌换人（PokéLLMon 失败模式）。
6. **精美加分**：宝可梦出场滑入动画、HP/种族值条填充动画、精灵球 loading、空态精灵球占位；每个动画浏览器实测。

**优先级**：P0（队伍视图/克制分析/种族值雷达）→ P1（对战模拟/进化链）→ P2（精修）。

**工具数量更新**：从 5 个增至 **6 个**（新增 `recommend_switch`）。

## Q8. 多轮对话设计 + 小模型兼容（队伍状态演进）

**问题**：多轮对话怎么设计？自主决定怎么兼容小模型？

**核心洞察**：B 的多轮是"修改持久状态"（换掉水系换电系 = 回忆队伍→识别水系→移除→找电系→更新，5 步链），不是 A 的"引用 id"。纯靠对话历史维护队伍对小模型必崩（F4+F5 叠加）。

**决策：演进 Q3——服务端结构化队伍 + 每轮注入 + 显式增删工具**（revises Q3 的纯对话历史方案）：
1. session 存结构化 `team: PokemonSnapshot[]`（≤6，含 id/name/nameZh/types/stats）。
2. **每轮把当前队伍注入上下文**（ephemeral 系统消息，按 state 重建，标好每只属性）→ 模型永远读真相不靠记忆。
3. 显式工具：`add_to_team` / `remove_from_team` / `replace_team_member`（高层一步替换）。
4. 变更后重算 → emit `team` 事件 → 前端更新 + 下轮重新注入。

**小模型兼容（沿用 A 四层防御 + B 特化）**：
- 第1层：6 工具按"查询/队伍/对战"分组说明；每轮注入队伍消除记忆负担。
- 第2层：宝可梦名静态词典 + PokeAPI 名单模糊匹配（皮卡丘→pikachu，拼错纠正）。
- 第3层：空结果 hint、超时重试、满 6 只友好拒绝。
- 第4层：`replace_team_member`/`recommend_switch` 把多步链压成单次调用；克制/换人评分全代码算（防 PokéLLMon 恐慌换人/属性诈骗）。
- **三招 B 特有**：每轮注入队伍（消除 F5）+ 高层工具（消除 F4）+ 静态 typechart 代码评分（消除克制幻觉）。

## Q9. 完成任务还需考虑的点（完整性分析）

**问题**：完成这个任务还有哪些地方需要考虑？

**🔴 两个已实测验证、会改设计的关键点**：
1. **PokeAPI 自带官方中文数据**（实测确认）：宝可梦名(皮卡丘)、属性(火)、招式(火焰拳)、图鉴描述都有官方 `zh-hans` 字段。
   - **彻底改变翻译策略**：题目 A 因 TheMealDB 无中文，折腾静态词典+LLM翻译+闪烁；**B 完全不需要**，规范化时直接取 `zh-hans` 字段，质量更高、零 LLM、零延迟、零闪烁。
   - 修正：之前方案里"宝可梦名静态词典+LLM翻译"作废，改为"提取官方中文字段"。静态词典只保留给"用户中文输入→英文 query"的反向映射。
2. **官方立绘图片在中国网络可能加载不出**：URL 是 `raw.githubusercontent.com`，国内常被墙/极慢。
   - 对策：后端图片代理 `/api/sprite/:id`（转发+缓存），或镜像，或前端失败 fallback 精灵球占位。

**🟠 性能与正确性**：
3. N+1 API 爆炸：组队触发大量调用，PokeAPI 要求本地缓存不高频轰炸 → cache.ts 必须够强。
4. typechart 世代与边界：用第 6 世代后现代克制表并标注；**必须处理 0× 免疫**（普通打幽灵、地面打飞行），双属性叠加 0× 会归零；18×18 必须单元测试全覆盖（一格错则全错）。
5. recommend_switch 诚实标注"启发式"：只用种族值+属性+速度，无 EV/IV/性格/道具/特性，不号称精确对战模拟。

**🟡 状态与工程**：
6. 队伍刷新恢复：队伍是服务端权威，需 `GET /api/team`（按 session）让前端重建；聊天历史走 localStorage（来源不同要理清）。
7. 队伍边界：重复宝可梦？空队伍提示？满 6 只拒绝？
8. 冷启动引导（快捷入口）、README（PokeAPI 署名+chiptune 许可证+截图）、.env/key 安全、提交不带 Claude 署名、移动端 Dashboard 折叠。
